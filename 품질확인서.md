# 품질확인서 — jeonse-pnu (v0.4.0)

작성일: 2026-06-30 · 검증 환경: Python 3.12.3
대상: **연립·다세대(VWorld API) + 오피스텔(국세청 파일 DB) 통합 파이프라인**

---

## 1. 서비스 범위 확정 (HUG 126% 룰)

전세보증 한도는 시세 없는 주택에서 **공시가격 × 적용비율(126%)**로 산정된다.
따라서 본 서비스 대상은 **시세가 없는 물건만**:

| 물건 | 한도 산정 | 데이터 소스 | 본 서비스 |
|---|---|---|---|
| 아파트 | KB시세 기반 | (공시가 안 씀) | 제외 |
| 연립·다세대(빌라) | 공시가격×126% | VWorld 공동주택가격 API | 포함 |
| 오피스텔 | 기준시가×126% | 국세청 기준시가 파일 | 포함 |

공동주택구분코드: 1=아파트(제외), 3=연립, 5=다세대 (정의서 R25 확정).

## 2. 통합 구조 (두 경로)

```
등기부주소 → 코어(juso 정제 → PNU 조립)
  → 두 경로 동시 조회 후 값이 나오는 쪽 채택:
     ├ 연립·다세대 → VWorld getApartHousingPriceAttr (실시간 API)
     └ 오피스텔     → 국세청 기준시가 DB (build_officetel_db.py 적재)
  → 구·신 공시가 + 물건종류 + 신뢰도
```

한 물건은 둘 중 하나에만 존재하므로 종류 판별 없이 둘 다 조회해 안전하게 채택.

## 3. 데이터로 확정한 사실 (전부 실파일 검증)

### 연립·다세대 (VWorld API)
- 공식 명세 확정: getApartHousingPriceAttr, 입력 pnu/stdrYear/dongNm/hoNm/key/domain
- 출력 19필드, pblntfPc=공시가격. 사용자 제공 실응답(상암월드컵1단지 6,000만원)으로 파싱 100% 검증
- PNU 단지 중복 → PNU+동+호 매칭. 다세대 동명 공백 → 호명만 매칭

### 오피스텔 (국세청 파일)
- 2025·2026 기준시가 xlsx 각 ~249만 행(오피스텔 ~133만). EUC-KR, 15컬럼
- 핵심 발견: 파일의 '호'(0005)는 지번 부번이고, 실제 호수는 '상가건물호주소'(201) 컬럼
- PNU = 법정동코드+산여부+번지(본번)+호(부번). 호 매칭키 = 호주소
- 부천 법정동코드는 41190 아닌 41192/41194/41196(구 단위) — 실 법정동 자료로 확인

# 추가 (v0.5.0) — 오피스텔 Turso(libSQL) 운영 연동

작성일: 2026-06-30

## 배경
Vercel 서버리스는 로컬 SQLite 파일을 호스팅하지 못한다(함수가 ephemeral).
따라서 오피스텔 DB는 운영에서 **Turso(libSQL)** 로 조회하도록 이원화했다.
연립·다세대는 VWorld 실시간 API라 DB 자체가 불필요(비대칭 유지).

## 구현
- officetel.py: sqlite3 + libSQL 자동 판별
  - 우선순위 TURSO_DATABASE_URL(+TOKEN) > OFFICETEL_DB_PATH
  - 조회 결과를 dict로 정규화해 두 백엔드 동일 처리
- scripts/push_to_turso.py: 로컬 officetel.db → Turso 배치 업로드(libsql_client.batch)
- requirements.txt: libsql-client>=0.3.1

## 검증 (Turso 네트워크 차단 환경에서의 등가 검증)
컨테이너에서 turso.tech 직접 연결은 차단되나, libsql-client는 `file:` URL로
**동일한 API**를 사용하므로 전 경로를 등가 검증:
- libSQL 조회 경로: file: URL 연결 → fetch_officetel_by_pnu(conn=client)
  → 인터시티오피스텔 201호 2,005,000원 정확 (R36)
- push_to_turso 업로드: 현대드림타워 901호 적재→업로드→libSQL 조회 값 일치(2,341,000원) (R37)
- 회귀: sqlite 경로(db_path/주입 커넥션) 기존 동작 보존 확인

실제 Turso는 URL만 libsql://...turso.io 로 바꾸면 동일 동작(API 동일).

## 검증 절차 결과
- syntax/import OK
- 테스트: 코어 12/12 + 파이프라인 9/9 = **21/21 통과** (libSQL 케이스 추가)
- pyflakes 0건 클린
- 프론트 vite build 성공(35모듈)

## 남은 것 (사용자 환경)
1. Turso DB 생성(turso.tech 또는 Vercel Marketplace Turso Cloud 연동)
2. 로컬 적재 → push_to_turso.py 로 업로드(전국 133만 행)
3. Vercel 환경변수: JUSO/VWORLD/VWORLD_DOMAIN/TURSO_DATABASE_URL/TURSO_AUTH_TOKEN
4. VWorld 라이브 키 호출 1회 확인(연립·다세대)

  프론트가 오피스텔 아니면 전부 "연립·다세대"로 하드코딩.

1. has_jibun을 등기부 파싱값(parsed.본번)으로 판단 → 도로명주소는 parsed.본번이
   비어 지번 15점이 안 붙음(도로명은 juso가 지번을 채우는데 그걸 반영 못함).
2. 도로명주소에서 당연한 안내("호 미인식","읍면동 미인식","도로명주소-지번 juso 정제")와
   중복 경고("동 구분 없어 호로만 특정" ×2)가 경고 감점(-3씩)으로 누적.
결과: 40+0(지번탈락)+28-15(가짜감점)=53 → C.

## 설계
1. has_jibun을 '정제 결과 기준'으로: geo.parts.본번 또는 PNU 확보 여부로 판단.
2. confidence 경고 감점에서 '정상 안내'는 면제(BENIGN 목록), 중복 경고는 1회만.

## 개발
- pipeline.py: has_jibun = bool(geo.parts.본번) or bool(out.pnu)
- confidence.py: BENIGN 필터 + seen 집합으로 중복 제거, 진짜 경고만 -3

## 검증 (5시나리오 + 전체흐름)
- 도로명+호매칭(센트레빌): 53→83점 B ✔ (핵심 수정)
- 완벽(1차+지번+호+등기교차): 100 A ✔
- 호 미확정(동만): 55 C ✔ (호 없으면 여전히 C — 정상)
- 폴백+호: 65 B ✔ / 정제실패: 0 F ✔
- 회귀: 코어 12/12 + 파이프라인 18/18(테스트 추가), pyflakes 클린, 빌드 성공

## 요구사항 적합성
동·호로 세대가 특정된 정상 조회가 부당하게 C로 떨어지던 문제 해결.
호 미확정 시 C(수동확인 유도)는 의도된 동작으로 유지. 적합.

## 신뢰도 등급 기준(참고)
A≥85(정제1차+지번+호+등기교차=100) / B≥65 / C≥40(호 미확정 등) / F<40(정제실패)
호(전유부분) 확정이 28점으로 가장 크며, 등기고유번호 교차검증 시 +17점으로 A 도달.

# 추가 (v0.7.6) — 건물명(아파트/오피스텔명)만으로 주소 정제 (고급 정제)

작성일: 2026-07-03

## 분석 (라이브: "구로구 진오피스텔" → 엉뚱한 구로구 동들 후보)
건물명만 입력하면 juso가 그 건물을 찾아 지번을 반환해야 하는데, 실제로는
"구로구"만 검색돼 고척동/구로동/가리봉동/항동이 후보로 뜸.
근본 원인: pipeline이 juso에 넘기는 검색어로 parsed.검색질의를 쓰는데,
registry_parser는 등기부 소재지번(행정구역+숫자지번) 파서라 건물명("진오피스텔")을
'지번 아님'으로 보고 떨궈버림. → "서울특별시 구로구 진오피스텔"이 "서울특별시 구로구"로 축소.
(검증: parse의 검색질의가 건물명 삭제 후 행정구역만 남김을 확인)

## 설계
juso는 건물명 검색을 지원하므로, 지번(본번)이 없으면 원문을 그대로 juso에 넘긴다.
- 본번 있음 → parsed.검색질의(상세 제거본) 사용
- 본번 없음 → 원문 사용(건물명 유지). 단, 별도 필드로 받은 동/호는 검색어에서 제거
  (건물명 검색을 '105 201'이 오염시키지 않도록 _strip_unit_tail).

## 개발
- pipeline.py: 검색어 결정 분기(본번 유무) + _strip_unit_tail 헬퍼 신설.

## 검증
- "서울특별시 구로구 진오피스텔" → juso 검색어에 '진오피스텔' 유지, PNU 조립 ✔
- 건물명 + 동/호(105/201) → 검색어는 건물명만(동/호 오염 없음) ✔
- 후보 대표주소 재조회 → 단일 확정(pickRegion 정상 동작 재확인) ✔
- 회귀: 코어 12/12 + 파이프라인 19/19(테스트 추가), pyflakes 클린, 빌드 성공

## 요구사항 적합성
"아파트/오피스텔 이름만으로도 주소가 좁혀져 나와야 고급 정제" 요구 충족.
건물명을 juso 검색에 살려 정제 성공률을 높임. 적합.

## 후보 클릭 무반응(문제2) 관련
pickRegion은 v0.7.3에서 후보 대표주소 재조회로 수정 완료, 본 검증에서 정상 확인.
라이브 무반응은 배포가 v0.7.3 미만인 탓 → 이 빌드 배포 시 해소.
또한 건물명 정제로 잘못된 동명이지 화면 자체가 감소.

# 추가 (v0.8.0) — 세대(동·호) 목록 표시 + 선택

작성일: 2026-07-03

## 분석 (사용자 요청)
여러 세대 단지에서 "동·호를 입력하라"고만 하고 어떤 동·호가 있는지 안 알려줌.
사용자는 '이 단지에 몇 동 몇 호가 있는지' 목록을 보고 고르길 원함.
(추가로 동·호 입력이 매칭 안 되던 현상 → VWorld가 동을 비워주는 단지 이슈는
v0.7.4에서 '동 없으면 호로만 매칭'으로 대응, 본 기능으로 목록 제공해 근본 개선)

## 설계
VWorld/오피스텔 응답의 세대 목록을 available_units로 반환하고, 프론트에서
동별 그룹핑 선택 UI 제공. 동이 있으면 동→호 2단계, 없으면 호만.
선택 시 동·호를 채우고 그 세대로 재조회. 임의 대표값은 계속 금지.

## 개발
- pipeline.py: LookupResult.available_units 필드, _unit_list() 헬퍼(중복제거+동→호 정렬),
  apt/officetel needs_unit 분기에서 목록 채움.
- App.jsx: unitList 상태, pickUnit 핸들러(동·호 채우고 재조회), UnitPicker 렌더 분기,
  StatusCard title 전달 누락도 수정.
- UnitPicker.jsx(신규): 동 그룹 선택 → 호 선택 그리드.
- app.css: 세대 선택 스타일.

## 검증 (백엔드 + DOM E2E)
- 백엔드: 4세대(101/102동 × 101/1403호) → available_units 4개, 동→호 정렬 ✔
- DOM: 동 목록(101동/102동) → 101동 클릭 → 호 목록(101·102·201·202·1403호) →
  1403호 클릭 → 결과(신개봉삼환) 렌더 ✔ (Playwright 확인)
- 회귀: 코어 12/12 + 파이프라인 20/20, pyflakes 클린, 빌드 성공

## 요구사항 적합성
"있는 동·호 목록을 보여주고 골라서 입력" 요구를 백엔드~프론트로 구현.
어떤 세대가 존재하는지 사용자가 직접 확인·선택 가능. 적합.
