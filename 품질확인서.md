# 품질확인서 — jeonse-pnu (v0.4.0)

작성일: 2026-06-30 · 검증 환경: Python 3.12.3
대상: **연립·다세대(VWorld API) + 오피스텔(국세청 파일 DB) 통합 파이프라인**

---

## 1. 서비스 범위 확정 (HUG 126% 룰)

전세보증 한도는 시세 없는 주택에서 **공시가격 × 적용비율(126%)**로 산정된다.
따라서 본 서비스 대상은 **시세가 없는 물건만**:

| 물건 | 한도 산정 | 데이터 소스 | 본 서비스 |
|---|---|---|---|
| 아파트 | KB시세 기반 | (공시가 안 씀) | 제외 |
| 연립·다세대(빌라) | 공시가격×126% | VWorld 공동주택가격 API | 포함 |
| 오피스텔 | 기준시가×126% | 국세청 기준시가 파일 | 포함 |

공동주택구분코드: 1=아파트(제외), 3=연립, 5=다세대 (정의서 R25 확정).

## 2. 통합 구조 (두 경로)

```
등기부주소 → 코어(juso 정제 → PNU 조립)
  → 두 경로 동시 조회 후 값이 나오는 쪽 채택:
     ├ 연립·다세대 → VWorld getApartHousingPriceAttr (실시간 API)
     └ 오피스텔     → 국세청 기준시가 DB (build_officetel_db.py 적재)
  → 구·신 공시가 + 물건종류 + 신뢰도
```

한 물건은 둘 중 하나에만 존재하므로 종류 판별 없이 둘 다 조회해 안전하게 채택.

## 3. 데이터로 확정한 사실 (전부 실파일 검증)

### 연립·다세대 (VWorld API)
- 공식 명세 확정: getApartHousingPriceAttr, 입력 pnu/stdrYear/dongNm/hoNm/key/domain
- 출력 19필드, pblntfPc=공시가격. 사용자 제공 실응답(상암월드컵1단지 6,000만원)으로 파싱 100% 검증
- PNU 단지 중복 → PNU+동+호 매칭. 다세대 동명 공백 → 호명만 매칭

### 오피스텔 (국세청 파일)
- 2025·2026 기준시가 xlsx 각 ~249만 행(오피스텔 ~133만). EUC-KR, 15컬럼
- 핵심 발견: 파일의 '호'(0005)는 지번 부번이고, 실제 호수는 '상가건물호주소'(201) 컬럼
- PNU = 법정동코드+산여부+번지(본번)+호(부번). 호 매칭키 = 호주소
- 부천 법정동코드는 41190 아닌 41192/41194/41196(구 단위) — 실 법정동 자료로 확인

# 추가 (v0.5.0) — 오피스텔 Turso(libSQL) 운영 연동

작성일: 2026-06-30

## 배경
Vercel 서버리스는 로컬 SQLite 파일을 호스팅하지 못한다(함수가 ephemeral).
따라서 오피스텔 DB는 운영에서 **Turso(libSQL)** 로 조회하도록 이원화했다.
연립·다세대는 VWorld 실시간 API라 DB 자체가 불필요(비대칭 유지).

## 구현
- officetel.py: sqlite3 + libSQL 자동 판별
  - 우선순위 TURSO_DATABASE_URL(+TOKEN) > OFFICETEL_DB_PATH
  - 조회 결과를 dict로 정규화해 두 백엔드 동일 처리
- scripts/push_to_turso.py: 로컬 officetel.db → Turso 배치 업로드(libsql_client.batch)
- requirements.txt: libsql-client>=0.3.1

## 검증 (Turso 네트워크 차단 환경에서의 등가 검증)
컨테이너에서 turso.tech 직접 연결은 차단되나, libsql-client는 `file:` URL로
**동일한 API**를 사용하므로 전 경로를 등가 검증:
- libSQL 조회 경로: file: URL 연결 → fetch_officetel_by_pnu(conn=client)
  → 인터시티오피스텔 201호 2,005,000원 정확 (R36)
- push_to_turso 업로드: 현대드림타워 901호 적재→업로드→libSQL 조회 값 일치(2,341,000원) (R37)
- 회귀: sqlite 경로(db_path/주입 커넥션) 기존 동작 보존 확인

실제 Turso는 URL만 libsql://...turso.io 로 바꾸면 동일 동작(API 동일).

## 검증 절차 결과
- syntax/import OK
- 테스트: 코어 12/12 + 파이프라인 9/9 = **21/21 통과** (libSQL 케이스 추가)
- pyflakes 0건 클린
- 프론트 vite build 성공(35모듈)

## 남은 것 (사용자 환경)
1. Turso DB 생성(turso.tech 또는 Vercel Marketplace Turso Cloud 연동)
2. 로컬 적재 → push_to_turso.py 로 업로드(전국 133만 행)
3. Vercel 환경변수: JUSO/VWORLD/VWORLD_DOMAIN/TURSO_DATABASE_URL/TURSO_AUTH_TOKEN
4. VWorld 라이브 키 호출 1회 확인(연립·다세대)

# 등기부 파서 2차 — 좌표 기반 표 복원 (v0.9.5)

작성일: 2026-07-12
검증 자료: 실제 등기부 PDF 2건 (사용자 제공)
  · PDF1 강남 상록수 203동 302호 — 을구 없음, 압류·가처분 전부 말소
  · PDF2 인천 계산동 더뷰스퀘어 101동 404호 — 근저당·신탁·강제경매·주택임차권 (HUG 사고물건)

## 1차 파서의 치명적 결함 (근저당 실물로 발견)

PDF2로 1차(텍스트 줄 기반) 파서를 검증하자 3가지 오답이 나왔다.

| 항목 | 1차 파서 | 정답 |
|---|---|---|
| 채권최고액 | 0원 | 3번 10,000,000원 |
| 갑구 10·11·12번 | 유효 위험(경매 진행중!) | 14번에서 전부 말소 |
| 갑구 14번 | "가압류" 위험 오탐 | 말소 이벤트 행 |

근본 원인 2가지:
1. PDF 하단 **안내문구** "기록사항 없는 갑구, 을구는 '기록사항 없음'으로 표시함"을
   실제 "을구 없음"으로 오판 → 근저당 파싱 자체를 건너뜀.
2. **등기부는 표(表)다.** 텍스트를 줄 단위로 읽으면 셀이 뒤섞인다. 실제 추출 순서:
     14 / "10번가압류," / 2025년7월7일 / "11번압류," / 제3514834호 /
     "12번강제경매개시결" / 인한 매각 / "정등기말소"
   등기목적 한 칸이 4줄로 쪼개지고 그 사이에 접수·등기원인이 끼어들며,
   단어 중간에서도 절단된다("강제경매개시결"+"정등기말소").
   → 줄 단위 파싱으로는 **원리적으로 복원 불가**.

## 근본 해결 — 좌표 기반 표 복원

- 등기부는 대법원 표준양식이라 컬럼 헤더 x좌표가 일정하다(실측: 두 PDF 오프셋 ~10px).
- 헤더 줄에서 x 간격 30px 이상을 컬럼 경계로 산출 → 각 아이템을 x로 컬럼 분류.
- 같은 y(±4px)를 '줄'로 묶어 섹션 마커(【갑구】/【을구】)·헤더를 판정.
  (PDF.js는 자간 넓은 글자를 낱개로 쪼갠다: '【','갑','구','】'. 줄로 묶어야 인식됨)
- 페이지가 바뀌어도 진행 중인 순위 블록을 유지(15번 신탁이 페이지 넘어 이어짐).
- 결과: `10번가압류, 11번압류, 12번강제경매개시결 정등기말소` 완벽 복원.

## 말소 판정 규칙 (실물 검증)

- 등기목적 칸에 "말소"가 있으면, **그 칸의 모든 "N번"**을 해당 섹션에서 말소 처리.
  → 복수 말소("10번…, 11번…, 12번…등기말소") 정확히 처리.
- 권리자 칸 본문의 "말소"는 무시. (을구 3-1 "3번근저당권공동담보소멸"의 본문에
  "근저당권말소등기로 인하여"가 있으나 이는 **다른 부동산**의 공동담보 소멸이며
  3번 근저당은 유효 → 오탐 없음을 실물로 확인)
- 도로명 "N번길" 오탐 방지(`(\d+)번(?!길)`).

## 2차에서 새로 가능해진 것

- **현재 소유자 추출** + **신탁 감지**: PDF2는 소유자가 대한토지신탁(수탁자) →
  "수탁자 동의 없이는 보증금을 지키기 어려움. 신탁원부 확인" 경고 표시.
- 근저당권자(금융기관명) 추출: 송림중앙신용협동조합.
- 임차권·전세권의 보증금액 추출.
- 섹션(갑구/을구) 구분 표시, 원문 근거(rows) 반환.

# 등기부 파서 2차 — 주요 등기사항 요약 파싱 + 교차검증 (v0.9.6)

작성일: 2026-07-12

## 왜 요약 페이지인가

발급본 뒤에 붙는 "주요 등기사항 요약"은 **말소되지 않은 사항만** 정리한 표다.
즉 말소 판정이 이미 끝난 정답지다. 본문(갑구·을구) 파싱 결과와 대조하면
파서가 스스로를 검증할 수 있다. (GitHub 실무도구 msson1120도 이 페이지를 파싱한다)

## 구현

- `parseSummary(items)` — 요약 페이지의 3개 섹션을 파싱한다.
  · §1 소유지분현황(갑구): 등기명의인 | 등록번호 | 최종지분 | 주소 | 순위번호
  · §2 소유권 제한사항(갑구), §3 (근)저당권·전세권(을구): 순위 | 등기목적 | 접수 | 주요등기사항 | 대상소유자
- 요약표는 **헤더가 중앙정렬, 데이터가 좌측정렬**이라 본문과 달리 헤더 x로 컬럼을 잡으면
  값이 옆 칸으로 밀린다(채권최고액이 접수 칸으로 감). → 데이터 x 분포로 컬럼을 산출.
- 브라우저(PDF.js)는 pymupdf와 텍스트 분할이 달라 컬럼 경계가 또 어긋났다.
  → **값 추출(금액·채권자·지분)은 행 원문(raw)에서 정규식으로** 뽑아 컬럼 분류에 의존하지 않게 수정.
  (컬럼은 행 구분에만 사용)

## 교차검증

본문 파싱과 요약 파싱을 대조한다.

| | 본문(좌표 표복원) | 요약 페이지 | 판정 |
|---|---|---|---|
| 채권최고액 | 10,000,000원 | 10,000,000원 | 일치 |
| 소유자 | 대한토지신탁주식회사 | 대한토지신탁주식회사 (수탁자) | 일치 |

두 경로가 독립적으로 같은 값을 냈다 → 본문 좌표 파서의 정확성을 뒷받침한다.
불일치 시 화면에 경고를 띄우고 원본 확인을 안내한다.
요약 페이지가 없는 발급본(PDF1)은 `crossCheck.available=false`로 본문 파싱만 사용한다.

## 검증

  · 교차검증 일치, 요약 없는 발급본 처리

기존 원칙은 "보증 한도 계산 안 함"이었다(선순위채권을 알 수 없어 오답 위험).
이번에 **선순위를 사용자에게 입력받는 것**으로 정리하면서 한도 계산이 가능해졌다.

## 조사 (추측 0)

HUG 전세보증금반환보증, 비아파트(연립·다세대) 기준:
```
주택가격 = 공시가격 × 140%   (1순위 가격 산정)
보증한도 = 주택가격 × 90%    (담보인정비율)
```

· **2026년 7월 현재 126% 유효.** 담보인정비율 80%(=112%) 하향은 검토만 되고 미시행

# 네이버 딥링크 — center 인코딩 역산 (v0.10.3)

작성일: 2026-07-12

## 역산 결과 (확정)

네이버 `center` 파라미터는 위경도를 base62로 인코딩한 값이다.

```
center = enc(경도 × 1e7 + 2e9) + "-" + enc(위도 × 1e7 + 2e9)
enc    = base62, 문자셋 "0-9 a-z A-Z", 6자리 좌측 0패딩
```
오프셋 2e9는 음수 좌표 제거용(경도 -180 → 2e8, +180 → 3.8e9. 6자 base62 최대 5.68e10 안에 들어감).

### 검증 과정 (실기기 왕복)
1. 네이버가 만든 `3zfQf6-2AHIg8` → 디코딩 → (37.4632, 126.9346) → 재인코딩 시 **완전 일치**.
2. 이 좌표는 스크린샷 지도 중심(관악구 남부)과 일치.
3. **1차 실기기 테스트에서 김포 3곳이 고양·파주·마포로 빗나갔다.**
   원인은 인코딩이 아니라 **필자가 URL을 손계산으로 만든 오류**였다.
   실제로 그 잘못된 center를 디코딩하니 네이버가 이동한 지점(고양 문봉동 등)과
   오차 40~150m로 일치 → **디코더가 정확함을 역으로 증명**.
4. 코드로 재생성한 김포 3곳: 전부 정확히 이동 확인.
5. 지방광역시 3곳 중 광주 치평동 정확 적중. 부산·대구가 옆 동으로 간 것은
   필자가 넣은 랜드마크 좌표가 동 경계에 걸린 탓(해운대역=우동/중동 경계).
   → **인코딩은 좌표를 정확히 옮긴다. 필요한 것은 정확한 법정동 좌표뿐.**

## 구현

- `src/utils/links.js` — `naverCenter()` / `readNaverCenter()` / `naverLandUrl(deposit, coord)`.
- `scripts/build_area_stats.py` — `--kakao-key` 옵션 추가.
  각 법정동을 카카오 주소검색으로 지오코딩해 `lat`/`lon`을 저장한다(캐시 지원).
  키가 없으면 좌표 없이 생성되고, 프론트는 클립보드 폴백으로 동작한다.
- `src/components/AreaFinder.jsx` — 좌표가 있으면 **그 동네로 바로 가는 딥링크**,
  없으면 필터만 걸고 동네명 복사.
