# ltvcheck 기능명세서 v2 (코드 기준 v0.9.0)

> 전세보증 도우미 — 공동주택·오피스텔 공시가격 조회 및 1억 이하 주택 탐색
> 작성: 2026-07-09 · 실코드(newwonwoo/ltvcheck) 대조 작성
> **이 문서는 What뿐 아니라 Why(설계 근거)와 알려진 결함까지 기술한다.**
> 리뷰어는 §3(설계 결정)과 §8(알려진 결함)을 먼저 읽을 것.

---

## 1. 서비스 정의와 범위

### 1-1. 하는 것
| 탭 | 기능 | 입력 | 출력 |
|---|---|---|---|
| 탭1 공시가 조회 | 특정 세대의 작년·올해 공시가 변동 | 주소(+동·호) | 구·신 공시가, 증감, 확인상태 |
| 탭2 1억 이하 주택 | 지역 내 공시가 1억 **이하** 주택 목록 | 지역+유형+면적 | 목록, ㎡당 공시가 |

### 1-2. 명시적으로 하지 않는 것 (의도적 범위 제외)
- **보증 한도 계산(공시가×126%)을 하지 않는다.** 한도 산정에는 선순위채권 차감,
  주택유형별 가격 적용 순서, 규정 예외 등이 얽혀 있어 숫자 하나로 내면 오답 위험.
  본 서비스는 "기준이 되는 가격의 변동"까지만 안내. (사용자 확정 결정)
- **투자 관점 정보(저평가/수익률 등)를 제공하지 않는다.** 공공기관 성격상 투기 조장
  오해 방지. 사실(공시가·면적·㎡당 단가)만 제공, 해석은 사용자 몫. (사용자 확정 결정)
- **아파트의 한도 안내를 하지 않는다.** 아파트 한도는 KB시세 기반이라 공시가 변동이
  한도에 직결되지 않음. 공시가 자체는 표시하되 "참고용 + 시세기준" 고지. (차단 아님)

### 1-3. 서비스명 관련 결정 대기
현 헤더는 "전세보증 도우미". "한도 미리보기"라는 옛 명칭은 한도 계산을 하지 않는
범위와 충돌하므로 폐기. 최종 명칭은 미정(후보: 전세보증 기준가격 조회).

## 2. 데이터 원천 (원천 → 접근 → 필드 → 가공 → 알려진 이슈)

### 2-A. 연립·다세대 공시가격 — VWorld 실시간 API

```
[원천]   국토교통부 공동주택 공시가격 (공시기준일 매년 1/1)
[접근]   GET https://api.vworld.kr/ned/data/getApartHousingPriceAttr
[요청]   pnu(19자리, 필수) · stdrYear · format=json · numOfRows=1000 · pageNo=1
         · key(VWORLD_API_KEY) · domain(조건부)
         ※ dongNm/hoNm 서버필터 파라미터도 존재하나 의도적으로 미사용 → §3-1
[응답]   apartHousingPrices.fields.field[]:
         pnu, aphusNm(단지명), aphusSeCode/Nm(1아파트/3연립/5다세대),
         dongNm, hoNm, floorNm, prvuseAr(전용㎡), pblntfPc(공시가격, 원), stdrYear
[가공]   전 세대 수신 → 클라이언트 매칭(§4-3) → 세대 특정 → 가격
[단위]   pblntfPc = 세대(전유부분) 전체 공시가격, 원 단위. ㎡당 아님. (실측 확인:
         센트레빌 84㎡ 세대 5.1억 — 총액으로 정합)
```

**알려진 이슈**
- (확인됨) **dongNm이 빈 값("")인 단지가 실존** — 센트레빌레우스1단지 실측.
  → 매칭 규칙 2단계(호로만 매칭)로 대응 완료(§4-3).
- (미해결, P0-7) numOfRows=1000 초과 대단지는 2페이지 이후 세대 누락.
  totalCount는 읽으나 페이지 순회 없음. → 페이지네이션 추가 예정.
- (미해결) 인증키가 개발키. 서버호출(Referer 없음) 시 INCORRECT_KEY 가능.
  `/api/vworld-diag`가 domain 3종 실호출로 통과 조합 판별. 운영키 전환 권장.
- (미해결, P0-5) stdrYear를 시스템 연도로 자동 설정 → 연초(1~4월) 신년도 미공개
  기간에 "데이터 없음"과 "미공개"를 구분 못 함. → 연도 폴백 설계 예정(§8).

### 2-B. 오피스텔 기준시가 — 국세청 파일 → DB

```
[원천]   국세청 상업용건물/오피스텔 기준시가 (연 1회 고시, xlsx)
[필드]   소재지, 상가종류코드(='오피스텔'만 적재), 동, 층, 호('상가건물호주소'가
         실제 호수; '호' 컬럼은 지번부번이므로 혼동 주의), 전용면적, 고시가격
[적재]   build_officetel_db.py: xlsx → PNU 조립 → officetel 테이블
[저장]   3-백엔드 자동판별: Postgres(POSTGRES_URL) > Turso > 로컬SQLite
[조회]   SELECT ... WHERE pnu=? AND year=? → 호 매칭(§4-3과 동일 규칙)
```

**해결된 이슈 — ✅ P0-1/P0-3 (v0.9.2에서 확정·수정)**
- (해결) **고시가격 = ㎡당 단가로 확정.** 국세청 공식(손택스: "호별 기준시가 =
  단위면적 기준시가 × 건물면적(전용+공유)") + 실데이터 20만건 통계로 이중 확인.
  실제 총액 = 고시가격 × (전용면적 + 공유면적). ★전용만이 아니라 공유 포함.
  조회 결과에 계산근거(price_calc: ㎡당·전용·공유·총면적·공식) 노출.
- (해결) PK를 (pnu,building,dong,floor,ho,year)로 확장 — 같은 PNU 다른 동/층/호
  덮어쓰기 방지(P0-3). 스키마에 share 컬럼 추가.
- (남음) 실DB 적재는 사용자 환경 필요. 기존 적재분 있으면 신규 스키마로 재적재.
- (미해결) 실DB 적재가 사용자 환경에서 아직 수행되지 않음(운영 미연결).

### 2-C. 주소 정제 — 행정안전부 juso API (+카카오 폴백)

```
[원천]   행안부 도로명주소 (keyword 검색 — 도로명/지번/건물명 모두 지원)
[응답]   admCd(법정동10), siNm/sggNm/emdNm, roadAddr, jibunAddr,
         lnbrMnnm(본번)/lnbrSlno(부번), mtYn(산), bdKdcd(1=공동주택), bdNm(건물명)
[가공]   캐스케이드 정제(§4-2) → PNU 4요소
[폴백]   카카오 로컬 API (juso 실패 시)
```

### 2-D. 1억 이하 주택 — 국토부 CSV → 사전가공 정적 JSON (비적재)

```
[원천]   국토교통부 공동주택 공시가격 전국 호별 CSV
         (data.go.kr, 약 1,558만 행, 연 1회 갱신, 무료·이용제한 없음.
          '24년부터 관리건축물대장PK 제공 → 건축물대장 연계 가능)
[가공]   scripts/build_low_price.py (1회 실행):
         ① 공시가 ≤ 1억 필터  ← ★비적재의 전제(§3-4). 현 코드는 '<'이며
            '이하(≤)'로 수정 확정(§8 결정사항)
         ② ㎡당공시가 = 공시가 ÷ 전용면적 사전계산
         ③ 법정동코드5(시군구)별 JSON 분할, 공시가 오름차순 정렬
         ④ 컬럼명 자동추정(회차별 명칭차 흡수) + --col-* 수동지정
[산출]   public/data/low/_index.json(지역목록) + {시군구코드5}.json
[서빙]   Vercel 정적 파일. 조회 시 해당 지역 파일 1개만 fetch. DB·서버·API 없음.
```

**알려진 이슈**
- (미검증) 시군구 파일 크기 — 밀집 지역(예: 강서구 다세대)은 수 MB 가능.
  빌드가 용량 리포트를 출력하므로 **실데이터 1회 실행 후 실측으로** 분할
  (법정동/유형/가격구간) 여부 결정. 사전 과설계는 하지 않음.
- (미반영) JSON에 데이터 계보 메타(dataset_year, generated_at, source_hash,
  row_count, price_condition) 미포함 → 추가 예정. 화면에도 기준연도·생성일 표시.

## 3. 핵심 설계 결정과 근거 (★리뷰어 필독)

### 3-1. 세대 조회는 "전 세대 통수신 + 클라이언트 매칭" (서버필터 의도적 미사용)

**결정**: VWorld 호출 시 dongNm/hoNm 서버필터 파라미터를 쓰지 않고, 그 PNU의
전 세대를 받아 우리 정규화(_norm)로 매칭한다. 코드에 파라미터 지원은 구현돼
있으나(_build_url) 호출부에서 의도적으로 넘기지 않는다.

**근거 3가지**:
1. **UX 전제** — 여러 세대일 때 "존재하는 동·호 목록 판"(UnitPicker)을 깔아
   사용자가 고르게 한다. 이 판은 전체 세대 응답이 있어야만 만들 수 있다.
   서버필터로 1건만 받으면 판 자체가 불가능.
2. **표기차 흡수** — 원천의 동·호 표기는 "제105동"/"105동"/"0105"/"B101" 등
   제각각. 서버필터는 문자열 일치라 표기가 다르면 0건(조용한 미매칭). 우리
   _norm은 이를 전부 동일시(§4-3).
3. **실측 사례** — 센트레빌레우스1단지는 dongNm이 전부 빈 값. 서버필터로
   dongNm=105를 보내면 0건이지만, 실제로는 호(1403)로 특정 가능했다.
   서버필터만 믿으면 "있는 세대를 없다"고 답하는 케이스가 실존.

**트레이드오프**: 응답이 큼(세대수만큼). 1,000건 초과 시 페이지네이션 필요(P0-7).
**기각된 대안**: 서버필터 우선 + 실패 시 전체조회 폴백(하이브리드) — 0건이
표기차인지 부재인지 구분 불가라 결국 폴백 호출이 흔해지고, 부분일치로 엉뚱한
1건이 오면 그대로 믿어버릴 위험. UX 전제(판)와도 불합치.

### 3-2. 임의 대표값 금지 (전 구간 일관 원칙)

**결정**: 특정하지 못하면 값을 만들어내지 않는다. units[0] 같은 폴백 전면 금지.
- 여러 세대 + 동·호 미특정 → 값 없음 + needs_unit + available_units(목록) 반환
- 세대가 정확히 1건 → 특정 불필요, 그 값 사용
- 오피스텔도 동일 규칙

**근거**: 세대마다 공시가가 다르다. 첫 세대값을 대표로 내면 "정확한 숫자처럼
보이는 오답"이 된다. 이 서비스에서 가장 위험한 실패 모드.

**과거 결함**: 초기엔 대표값 폴백이 있었고(F등급으로 완화 표시), 사용자 지적으로
전면 제거(v0.7.0). "신뢰도 낮음 표시"는 오답의 면죄부가 되지 못한다.

### 3-3. 주소 모호성은 숨기지 않고 노출

**결정**: 동명이지(同名異地)·다중 후보는 1순위 자동확정하지 않고 후보 전체를
반환해 사용자가 고른다(RegionPicker).

**정밀 조건(수정 예정 포함)**:
- 같은 필지(같은 PNU)의 복수 표현(도로명/지번 등) → 압축 확정 (정당)
- **다른 PNU가 2개 이상 → 반드시 노출** ← 현 코드는 법정동코드 기준 그룹핑이라
  같은 법정동 내 다른 번지(737 vs 737-1)를 첫 후보로 자동확정하는 결함(P0-2).
  그룹키를 법정동코드 → PNU 전체(법정동+산+본번+부번)로 수정 예정.
  같은 법정동이라도 PNU가 다르면 완전히 다른 물건이다.

### 3-4. 탭2는 비적재(No-DB) — "상한 고정"이 전제조건

**결정**: 1,558만 행을 DB에 넣지 않는다. 상한(1억)이 서비스명에 고정돼 있으므로
사전 필터로 데이터가 수십 KB~수 MB로 줄고, 정적 JSON 서빙으로 충분하다.

**근거**: "지역+가격조건 → 목록"을 주는 무료 실시간 API가 존재하지 않음을 조사로
확정(VWorld=PNU 단건, WFS=좌표bbox·속성필터 불가, CODEF=유료 배제). 남은 선택지는
"어딘가에 두고 쿼리"뿐이며, 상한 고정 덕에 가장 가벼운 형태(정적 파일)가 가능.

**확장 규칙**: 보증한도 탐색도 같은 엔진 — 전세보증 상한(수도권 7억, 공시가
역산 약 5.6억)이 있으므로 --limit만 바꿔 동일 구조로 생성 가능. build 스크립트는
이미 상한 파라미터화됨.

### 3-5. 아파트는 "표시하되 구분" (차단도, 동일취급도 아님)

**결정**: VWorld 응답의 aphusSeCodeNm으로 실제 유형을 판정. 아파트면 공시가는
그대로 보여주되 배지("아파트·참고용")와 고지("한도는 KB시세 기준")를 붙인다.
is_target=False로 API에도 구분 노출.

**과거 결함 2건**: ①유형을 하드코딩("오피스텔 아니면 연립·다세대")해 아파트를
연립·다세대로 오표시 → 원천 필드 기반 판정으로 수정. ②아파트를 값까지 차단
→ "공시가는 존재하므로 보여준다"로 정정(사용자 피드백).

### 3-6. 신뢰도 점수 → 확인단계 표시로 전환 (확정, 미구현)

**현행**: 배점 합산(1차정제40+지번15+호28+등기교차17=100), A/B/C/F 등급.
**결함**: ①87점이 확률처럼 읽히나 경험적 가중치일 뿐(통계 보정 없음).
②has_ho가 `matched or bool(입력값)`이라 존재하지 않는 호를 입력해도 28점
가산(P0-4). ③단일 세대 건물에 틀린 호 입력 시 모순 미검출.
**전환안(사용자 확정)**: 점수 폐지, 단계별 통과/실패 + **실패 사유 명기**:
```
✅ 주소 확인   (PNU 확정)
✅ 건물 확인   (단지 식별)
✅ 세대 확인   (동·호가 원천 데이터와 일치)
❌ 전년도 비교 — 2025년 자료에 이 세대 없음
❌ 등기 교차검증 — 등기고유번호 미입력
```
통과는 간결히, 실패는 "무엇이 왜 안 됐는지"까지. 세대 확인은 반드시
`matched is not None` 기준(입력 존재가 아니라 원천 일치).

## 4. 백엔드 로직 상세 (jeonse_pnu)

### 4-1. PNU (pnu.py)
```
19자리 = 법정동코드(10: 시도2+시군구3+읍면동3+리2) + 산여부(1: 1일반/2산)
        + 본번(4, zfill) + 부번(4, zfill)
build_pnu() / split_pnu() 상호 검증됨.
예: 5121010200105960000 → 강원특별자치도 속초시 동명동 596
    (동명동 확정 근거: 구 강원코드 4221010200 실사례, 시도만 42→51 전환)
※ PNU는 필지(땅)까지만 식별. 동·호는 PNU에 없다 — 세대 식별은 원천 데이터의
  dongNm/hoNm(또는 등기고유번호)의 몫. 등기고유번호(법원, 13자리)와 PNU는
  직접 변환 불가, 다리는 '소재지번+동·호'뿐.
```

### 4-2. 정제 캐스케이드 (providers.py)
```
geocode(query):
  1) 원문 juso 검색 (tier=1) — juso는 건물명 검색 지원
  2) 동·호 제거본 juso 재검색
  3) 카카오 폴백 (tier=2)
  4) 뒤 토큰 절삭 재검색 — 단 ★도로명주소는 절삭 금지
     (근거: "도신로29길 28"에서 28을 떼면 juso가 '도신로'의 다른 길들
      [4길/48길/15길]의 29번지를 반환 — 실제 발생했던 오후보 사고)
전처리(registry_parser):
  - 도로명 내부공백 정규화: "도신로 29길"→"도신로29길" (건물번호 보존.
    미정규화 시 파서가 '29길'을 '29'로 오인해 건물번호 소실 — 실사고)
  - ★건물명 입력 지원: 지번(본번)이 없으면 파서 검색질의 대신 원문을 juso에
    넘긴다. (파서는 등기부 소재지번 전용이라 건물명을 '지번 아님'으로 버렸음
    — "구로구 진오피스텔"이 "구로구"로 축소돼 엉뚱한 동 후보가 뜬 실사고.
    별도 입력된 동·호는 검색어 오염 방지 위해 제거[_strip_unit_tail])
동명이지:
  - juso 후보를 그룹핑 → 여러 행정구역이면 ambiguous + region_candidates 전체 반환
  - 현 그룹키는 법정동코드(10) → P0-2에 따라 PNU 전체로 수정 예정(§3-3)
  - 후보 선택 시 프론트는 후보의 '대표주소'로 재조회 (원입력 재검색 시 같은
    후보가 무한 반복되는 실사고 → v0.7.3 수정)
```

### 4-3. 세대 매칭 규칙 (gongsiga.py — 오피스텔도 동일 원칙)
```
입력 정규화 _norm (숫자만 입력 인식):
  "제105동"="105동"="105", "0105"="105"(앞0 제거), "B101"="b101"→"101",
  숫자 없는 동(가동/나동)은 원문 소문자. gongsiga·officetel이 동일 함수 공유.

매칭 단계 (임의 대표값 금지):
  1) 동+호 모두 일치 → 그 세대
  2) 못 좁혔고 원천에 동 정보가 전무(전 세대 dongNm 빈값) → 호로만 매칭
     (센트레빌 실사례 대응. "동 데이터 없음"과 "동 특정 필요"를 구분)
  3) 호로만 좁혔는데 같은 호가 여러 동에 존재 → needs_unit (이때만 동 필수)
  4) 매칭 없고 세대 정확히 1건 → 그 값 (특정 불필요)
     ※ 알려진 구멍: 이 경우 입력 호와 원천 호의 모순을 검사하지 않음 —
       틀린 호를 넣어도 그 1건을 반환. P0-4와 함께 수정 예정.
  5) 여러 세대 + 미특정 → 값 없음 + needs_unit + available_units(동·호 목록,
     동→호 숫자정렬, 중복제거) 반환 → 프론트 UnitPicker가 선택 판을 깖.
```

### 4-4. 파이프라인 (pipeline.py)
